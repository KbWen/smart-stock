# APP Development Onboarding Guide

> How to use AgentCortex as your AI-governed development brain for APP projects.

## What AgentCortex Provides vs. What You Define

| AgentCortex (Brain) | Your Project (Downstream) |
|---|---|
| Workflow governance (Bootstrap → Ship) | Tech stack choices (React? FastAPI? PostgreSQL?) |
| Security scanning (OWASP + Red Team) | Domain-specific API conventions |
| Token management (context budget) | Frontend component patterns |
| Documentation lifecycle (Spec → ADR) | Database schema conventions |
| Evidence requirements (test gates) | Auth flow architecture |
| Multi-session continuity (handoff) | Business logic & domain rules |

**Key insight**: AgentCortex is the "how to develop safely" layer. Your project defines the "what to build and with which tools" layer.

## Getting Started (First-Time Setup)

### Step 1: Deploy Brain to Your Project

```bash
# From the brain template repo
./deploy_brain.sh /path/to/your/app-project
cd /path/to/your/app-project
./.agentcortex/bin/validate.sh
```

### Step 2: Start Your First Conversation

When you start your first non-trivial task (feature or architecture-change), the AI will detect that no project ADR exists and offer to run `/app-init`:

```
🏗️ New project detected — no architecture ADR found.
Run /app-init to establish project conventions? (yes/skip)
```

Say **yes**. The AI will ask about your tech stack and generate:
- An ADR documenting your architecture decisions
- Domain skills (api-design, frontend-patterns, database-design, auth-security) customized for your stack
- A spec template for your features

### Step 3: Normal Development

From this point forward, every feature follows the standard flow:
1. Describe your feature (natural language is fine)
2. AI runs `/spec-intake` → generates spec using your APP template
3. AI runs `/bootstrap` → classifies and loads context
4. `/plan` → `/implement` → `/review` → `/test` → `/ship`

Your domain skills are automatically recommended during bootstrap and loaded during implement/review.

## Mid-Project Architecture Changes

You don't have to decide everything upfront. Common scenarios:

### "I need to add a backend now"
Just tell the AI: "我要加後端" or "set up API layer". It will:
1. Read your existing ADR
2. Ask only about the new layer (backend framework, API style, etc.)
3. Update the ADR
4. Generate the `api-design` skill if it doesn't exist

### "I want to change the database"
Tell the AI: "改資料庫" or "switch to PostgreSQL". It will:
1. Create a new ADR that supersedes the old one
2. Update the `database-design` skill
3. Flag any existing specs that reference the old DB conventions

### "I need a custom skill"
Tell the AI: "新增 skill for [X]". It will generate a new skill scaffold that you can customize.

## Token Budget Impact

The APP development additions are designed for zero overhead on small tasks:

| Classification | Extra Cost from APP Features |
|---|---|
| **tiny-fix** | 0 tokens (all APP checks skipped) |
| **quick-win** | 0 tokens (all APP checks skipped) |
| **hotfix** | 0 tokens (unless touching auth — then auth-security skill loaded) |
| **feature** | ~50 tokens (ADR header check) + skill reads only when recommended |
| **architecture-change** | ~50 tokens (ADR header check) + skill reads + ADR full read |

Skills are read ONLY when recommended in the Work Log (during /implement), not during every bootstrap.

## FAQ

**Q: What if I already have my own coding conventions?**
A: Great — edit the generated SKILL.md files to match your conventions. They're scaffolds, not laws.

**Q: Can I use this with a monorepo?**
A: Yes. The ADR template supports monorepo project types. Each sub-project can have its own spec template variant.

**Q: What if I don't know my tech stack yet?**
A: Mark sections as `[TBD]` during /app-init. The AI will use generic conventions for those areas and prompt you to decide when a relevant task comes up.

**Q: Does this work with non-web projects (CLI tools, libraries)?**
A: The APP skills (API, frontend, DB, auth) are optional. If your project doesn't have a frontend, that skill simply won't be generated or recommended. The core governance framework works for any project type.

**Q: How do I update a skill after it's been generated?**
A: Edit the SKILL.md file directly, or tell the AI "update api-design skill" with your new conventions. The AI will merge your changes.
